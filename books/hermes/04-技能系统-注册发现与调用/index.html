四、技能系统：注册、发现与调用

技能是 Hermes 特有的可复用任务知识库——存储在 ~/.hermes/skills/ 目录下的 Markdown 文件，记录了完成特定任务的步骤、规范和注意事项。Agent 会将其以”用户消息”的形式注入对话，LLM 按照技能描述执行任务。

与工具的最大区别：工具是代码，技能是知识。工具执行确定性操作，技能引导 LLM 执行有判断力的操作。更重要的是，Agent 可以自己创建技**能**——在解决一个复杂问题后，把经验写成技能文件，下次遇到同类问题直接复用。

技能目录结构

SKILL.md 格式（YAML Frontmatter + Markdown 正文）：

渐进式披露：L0 静态注入 + L1/L2 运行时工具调用

这是技能系统最关键的设计决策。L0 是 prompt_builder.py 在构建 system prompt 时的静态写入行为（session 初始化时执行，仅含 name+description 摘要，约 50-200 token/条）；L1 是 LLM 运行时主动调用 skill_view(name) 加载完整 SKILL.md（约 500-2000 token）；L2 是 LLM 进一步调用 skill_view(name, file_path=...) 加载支持文件。三者性质不同，不应混为一谈。

    subgraph L1["Level 1：LLM 决定使用时（按需加载）"]

对比数据（10 个技能的场景）：

方案 token 消耗 说明
全量注入 system prompt ~20,000 token 每次请求都全量消耗，大部分无关
渐进式（当前设计） 5002,000 token 只加载实际用到的技能，节省 90%+

💡 关于 **~20,000 token 的估算**：代表性参考值，假设每个技能全文平均 ~2000 token（约 1500 字）。实际差异较大——简单工作流可能只有 200 token，详细多阶段教程可能超过 5000 token。该数值旨在说明量级差距，不是精确测量。

💡 会话中途安装新技能，何时生效？ L0 的技能摘要索引在 _build_system_prompt() 时一次性写入并缓存在 _cached_system_prompt，同一会话内不再重建（保持 prompt cache 稳定的核心约束）。因此会话中途安装的新技能，要到下一个新会话（或上下文压缩触发后）才会出现在 system prompt 里。上下文压缩会调用 _invalidate_system_prompt() 使缓存失效，再重调 _build_system_prompt()，此时新技能会被纳入。

显示调用某个技能时，为什么技能注入为”用户消息”而非”system prompt”

这是一个隐藏但极其重要的设计决策，与 Prompt Caching 深度耦合。凡是动态内容（会随场景变化）都注入到消息列表里；只有稳定内容（整个会话不变）才放入 system prompt。

sequenceDiagram
    participant User as 用户
    participant CLI as CLI
    participant Agent as AIAgent
    participant Anthropic as Anthropic API

    Note over Agent: system = "身份+记忆+技能索引"（稳定不变）

    User->>CLI: /github-pr-workflow 帮我创建 PR

    CLI->>Agent: messages = [\n  {"role":"user", "content":""}\n  {"role":"user", "content":"帮我创建 PR"}\n]
    Note over Agent: system 前缀没有变化！

    Agent->>Anthropic: {system: "...", messages: [...]}
    Anthropic-->>Agent: ✅ prefix cache 命中\nsystem token 费用 -90%

    Note over User: 如果把技能内容放进 system prompt：
    Note over Anthropic: ❌ system 前缀变了\ncache 失效，全量计费

实现细节：注入消息的 role 虽为 user，但内容以 [SYSTEM: The user has invoked the "xxx" skill, indicating they want you to follow its instructions.] 开头——刻意设计为 role: user 是为了不改变 system prompt 前缀，从而保住 prompt cache 命中（如果把技能内容放进 system prompt，前缀变化会导致缓存失效、全量计费）。注入时机：用户触发技能斜杠命令（如 /gif-search）后，CLI 的 build_skill_invocation_message() 构建出完整的消息字符串，通过 _pending_input 队列传给 process_loop，随后由 process_loop 作为独立的完整用户消息调用 chat()——技能内容是整条独立用户消息，而非某条用户消息的前缀。该消息经 run_conversation() 处理后进入 conversation_history。

💡 技能注入内容会被上下文压缩吗？ 技能以 role: user 消息注入对话历史，上下文压缩触发时这些消息会被纳入压缩摘要。这不会造成功能问题：压缩后 LLM 在 system prompt 里仍能看到技能的摘要索引，需要执行技能时仍会调用 skill_view 重新加载完整内容。L0（system prompt 摘要）→ L1（skill_view 按需加载）的渐进式机制使技能调用对压缩具有天然弹性。

Agent 自动创建技能（两种触发时机）

这是 Hermes “越用越聪明”的核心机制。技能来源共三种：来源 1 & 2 是 Agent 自动创建（两种触发时机见下方）；来源 3 是用户手动安装，通过 hermes skills install <name> 或从 agentskills.io 市场下载后放入 ~/.hermes/skills/。

💡 两种路径的本质区别：执行线程——主判断在主线程同步执行，Nudge 在独立后台线程异步执行；触发时机——主判断在任务完成瞬间，Nudge 在对话交付后；决策主体——主判断由主 agent 以亲历者视角决策，Nudge 由新建 review agent 以旁观者视角复盘；额外 LLM 调用——主判断为 0（只是多一次工具调用），Nudge 为 1+ 次（review agent 的完整对话，max_iterations=8）；两种路径都可拒绝保存（LLM 可忽略指令，review agent 可说”Nothing to save.”）。

技能的自我修复

当 LLM 使用技能（skill_view 加载后）发现内容过时或不完整时，会在当前对话轮次内直接 patch——SKILLS_GUIDANCE 指令要求：”patch it immediately — don’t wait to be asked”。自我修复发生在”阅读技能内容之后、执行任务期间”，不在 Nudge 触发时。

💡 write_file / remove_file 的使用场景：技能的 SKILL.md 是主入口，复杂技能还可能有支持文件（references/api-guide.md、templates/commit-message.md 等），通过 skill_view(name, file_path=...) 按需加载（L2 层）。write_file 用于创建/更新这些支持文件，remove_file 用于删除不再需要的支持文件。允许操作的子目录在源码中硬编码为 ALLOWED_SUBDIRS = {"references", "templates", "scripts", "assets"}，其他路径会被拒绝。
