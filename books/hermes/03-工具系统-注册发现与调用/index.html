三、工具系统：注册、发现与调用

Hermes 的工具系统基于一个简单但强大的设计原则：工具就是”**副作用导入**”——import 一个模块，工具就自动注册好了。没有集中式的工具配置文件，没有显式的初始化调用，一切在模块加载时自动完成。

这意味着添加一个新工具只需要创建一个文件，在文件末尾调用 registry.register()，然后在 model_tools.py 的导入列表里加一行。仅此而已。

工具过滤：按场景只暴露所需工具

注意

_HERMES_CORE_TOOLS 是 CLI 平台默认启用的工具名列表，不代表这些工具的 toolset 字段都叫 core。例如 execute_code 的 toolset 是 code_execution，browser_navigate 的 toolset 是 browser。check_fn 失败时该工具会被跳过，跳过日志为 logger.debug 级别，正常运行时终端不可见。

并行判断规则

条件 结果
批次只有 1 个工具 串行
含 clarify 强制串行
JSON 参数解析失败（args 非合法 JSON） 强制串行
全是 _PARALLEL_SAFE_TOOLS 白名单工具（read_file/web_search/web_extract/vision_analyze/session_search/skill_view/skills_list/ha_* 等） 并行（线程数 = min(工具数, 8)）
write_file/patch，且目标路径互不重叠 并行
其他工具（不在白名单且不是路径作用域工具） 串行

Agent 内部工具（不经过 registry）

_invoke_tool() 在调用 handle_function_call() 前，会先拦截以下工具直接处理（因为它们依赖 Agent 级状态）：

  • todo — 需要 TodoStore 实例
  • memory — 需要 MemoryStore 实例
  • session_search — 需要 SessionDB 实例
  • clarify — 需要 clarify_callback 回调
  • delegate_task — 需要 parent_agent 引用
  • memory_manager 插件工具 — 需要 MemoryManager 实例

只有这些都不匹配时，才转发到 handle_function_call() → registry.dispatch()。

_run_async() 的使用者：注册了 is_async=True 的工具

_run_async() 不是通用桥接——它只服务于 handler 本身是 **`async def` 的工具**。注册时声明 is_async=True，registry.dispatch() 会自动调用它。目前只有 4 类工具使用：vision_analyze（httpx 异步请求）、web_extract（并发抓取多 URL）、mixture_of_agents（并发调用多 LLM）、RL 训练工具 × 10（Atropos 异步环境）。

MCP 工具注册时 **`is_async=False**：MCP 有自己专属后台 event loop（_mcp_loop），handler 内部通过_run_on_mcp_loop()自行处理 async，对外暴露同步接口，不需要_run_async()`。

工具结果的大输出保护

工具可能返回巨大的内容（如读取一个 1MB 的文件），直接塞入 messages 会使上下文爆炸。Hermes 采用 3 层保护机制（registry.dispatch() 本身不做截断）：

层次 触发位置 机制
第 1 层 工具内部 各工具自己在返回前预截断（如 search_files）注册时有配置项max_result_size_chars
第 2 层 maybe_persist_tool_result() 单个结果超过工具注册的 max_result_size_chars 阈值 → 全文写入沙箱临时文件，context 里只留摘要 + 文件路径（模型可用 read_file 读全文）
第 3 层 enforce_turn_budget() 同一轮所有工具结果合计超过 200K chars → 把最大的未持久化结果溢出到磁盘

这套机制比简单截断更友好：内容不丢失，模型可以主动读取完整结果。

常见工具的 max_result_size_chars 参考值：

工具 max_result_size_chars 说明
read_file 200,000 约 50K tokens
web_extract 100,000 网页内容可能很长
terminal 50,000 命令输出
web_search 20,000 搜索结果摘要

CLI vs Gateway：为什么 event loop 的存在与否不同？

根本原因在于程序入口——CLI 以同步 main() 启动，进程里没有 event loop；Gateway 以 asyncio.run(start_gateway()) 启动，整个进程在一个持续运行的 loop 中。这决定了 _run_async() 走不同的分支。

CLI 路径：主线程直接阻塞（无新线程）
Gateway 路径：两层线程嵌套

两层线程的必要性：第 1 层（run_in_executor）防止 Agent 同步阻塞 Gateway 主 loop；第 2 层（_run_async 新线程）规避”在 running loop 里调 run_until_complete 会报 RuntimeError”的 Python 限制。CLI 无 running loop，直接 run_until_complete() 即可，无需新线程。

MCP 工具：连接外部服务器的动态工具

MCP（Model Context Protocol）是 Hermes 的”开放接口”——任何符合 MCP 协议的外部服务器（文件系统、GitHub、数据库等）都可以把它的工具动态注入到 Hermes，让 LLM 像调用内置工具一样调用它们。

MCP 关键设计细节

特性 实现方式
线程安全 _lock（threading.Lock）保护 _servers/_mcp_loop/_stdio_pids 的所有读写
异步隔离 专用后台 event loop（非主线程），通过 run_coroutine_threadsafe() 跨线程调用
断线重连 指数退避，最多 5 次，最长间隔 60s；第一次连接失败直接报错，不重试
安全过滤 stdio 子进程仅传白名单环境变量（PATH/HOME 等），防止泄露 API key
凭证脱敏 返回 LLM 的错误消息中，token/key/password 等凭证替换为 [REDACTED]
工具幂等 discover_mcp_tools() 是幂等的——已连接的 server 跳过，只连新增的
热重载 CLI /mcp reload 命令：先 shutdown_mcp_servers()，再 discover_mcp_tools()，最后刷新 agent.tools

工具命名规则

MCP 工具名格式为 mcp_{server_name}_{tool_name}，连字符和点替换为下划线。例如：server filesystem，tool read_file → mcp_filesystem_read_file；server my-api，tool fetch.data → mcp_my_api_fetch_data。

Plugin 系统：用户可扩展的工具与生命周期钩子

Plugin 系统是 Hermes 的用户扩展机制。与 MCP”连接外部服务器”不同，Plugin 是本地 Python 代码，可以注册新工具、注册生命周期钩子回调，甚至注入新的 CLI 子命令。

Plugin 的三种来源

来源 目录 启用方式
用户 Plugin ~/.hermes/plugins/<name>/ 目录存在即生效
项目 Plugin ./.hermes/plugins/<name>/ 需设置 HERMES_ENABLE_PROJECT_PLUGINS=1
Pip Plugin Python 包 entry-point hermes_agent.plugins pip install 后自动发现

Plugin 可注册的生命周期钩子

钩子名 触发时机 典型用途
pre_tool_call 工具调用前 记录审计日志、参数校验
post_tool_call 工具调用后 结果过滤、统计 metrics
pre_llm_call 每次调用 LLM 前 注入动态上下文（记忆召回），返回值会追加到 user message

on_session_finalize 会话持久化完成后 向外部系统同步会话数据
on_session_reset 会话被重置（/reset）时 清空插件维护的会话级缓存

重要：钩子注入不破坏 Prompt Cache

pre_llm_call 钩子的返回值只注入到 user message，永远不修改 system prompt。这是刻意设计：system prompt 在会话内冻结，修改它会导致 Anthropic prompt cache 失效，大幅增加 API 成本。所有插件提供的动态上下文都是临时的，不持久化到 session DB。

MCP vs Plugin：功能对比

维度 MCP 工具 Plugin 工具
实现语言 任意（Node.js、Python、Go…） 仅 Python
运行位置 独立进程或远程 HTTP 同进程内
连接方式 stdio 管道 或 HTTP 长连接 直接函数调用
生命周期钩子 不支持 支持（8 个钩子点）
热更新 notifications/tools/list_changed 动态刷新 不支持（需重启进程）
CLI 命令注册 不支持 支持（ctx.register_cli_command）
消息注入 不支持 支持（ctx.inject_message，可从外部触发对话）
适合场景 集成已有服务（文件系统、GitHub、数据库） 扩展 Agent 行为（记忆、审计、自定义工具）
