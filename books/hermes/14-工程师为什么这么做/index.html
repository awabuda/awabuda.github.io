十四、工程师为什么这么做

Hermes Agent 核心实现：工程师为什么这么做

本文不列功能清单，只回答一个问题：面对具体的**工程约束**，Hermes 的作者为什么做了这个选择而不是那个选择。每一节先给结论，再讲问题背景，再讲实现细节。

一、为什么 AIAgent 是同步的，明明 Python 有 async

结论：agent 主循环天然是串行的，async 在这里不带来任何并发收益，只增加复杂度和 bug 风险。

先想清楚一件事：LLM agent 的核心循环长这样——

这个循环每一步都必须等上一步完成，不存在可以”同时做”的工作。async/await 的价值在于”等 IO 的时候去做别的事”，但这里根本没有”别的事”可做，因为下一步的输入依赖上一步的输出。

但 Hermes 还有另一个约束：它需要同时支持 CLI 和 Gateway 两种入口。

CLI 是 Python 主线程，没有 event loop，代码里大量用了 subprocess.run()、sqlite3、os.path.exists() 这类同步阻塞 API。Gateway 是 asyncio event loop 驱动，消息通过 await 接收，agent 在独立线程里运行。

如果 AIAgent 是 async 的：

  • CLI 需要用 asyncio.run() 包一层，但 asyncio.run() 内部不能调用同步阻塞函数——那会阻塞唯一的 event loop，终端卡死
  • 要解决这个问题就要把所有工具改成 async，工作量巨大，且同步 subprocess 的 async 化本质上只是 loop.run_in_executor()，还是线程池，绕了一圈回到原点

Hermes 的解法是：编排层保持同步，只在真正需要 async 的地方做桥接。

路径B 为什么不用 asyncio.run() 而是持久 loop？因为 asyncio.run() 每次都创建然后销毁一个 event loop。httpx、AsyncOpenAI 这类异步客户端在 GC 阶段清理时会尝试调用 loop，发现 loop 已关闭，抛出 "Event loop is closed" 警告甚至异常。持久 loop 通过 threading.local() 绑定到当前线程，创建一次复用到底。

这个设计带来了一个清晰的结构：**_run_async()**** 是全仓库 sync→async 调用的唯一入口**，Browser tool（Playwright）、MCP tool 的异步调用全都走这一个函数。新增异步工具只要调用 _run_async()，不需要了解自己运行在哪个上下文。

二、为什么 run_agent.py 有上万行而不拆分成多个文件

结论：这不是懒，是刻意的。拆分会让”哪段代码控制哪个状态”变得模糊，出 bug 时更难追踪。

AIAgent.__init__() 有 45+ 个参数，初始化代码超过 700 行，整个文件将近 12,000 行。第一眼看很反直觉，但作者的逻辑很清楚：

agent 的主循环需要访问极其多的状态——当前的 token 使用量、iteration budget、conversation history、压缩器实例、memory store、工具列表、callback 集合、provider fallback 链……这些状态如果分散在多个类和文件里，每个方法调用都要跨边界传参或持有引用，任何一个状态不同步就会产生 bug，而且调试时你要在多个文件之间跳来跳去才能还原完整状态。

把所有东西放在一个类里，所有方法都直接访问 self，状态是单一的、可见的。代码量虽大，但每个状态在哪、被谁修改，是可以通过搜索直接找到的。

已经从 run_agent.py 拆出去的模块都有共同特征：它们的逻辑是相对独立的，和主循环之间的接口是清晰的。比如 prompt_builder.py（组装 system prompt）、context_compressor.py（压缩算法）、prompt_caching.py（Anthropic cache 标记）。这些模块接收输入、返回结果，不持有主循环的状态。

留在 run_agent.py 里的是那些”本质上是主循环的一部分”的代码——fallback 切换、budget 耗尽处理、工具执行的分发逻辑。这些代码如果拆出去，拆出去的类就需要持有 AIAgent 实例的引用，等于变相绕了一圈。

三、IterationBudget 的三个设计细节：为什么这么不简单

结论：一个看似只是”计数器”的类，解决了三个不同的工程问题。

表面上 IterationBudget 只是个带 lock 的计数器。但细看实现，每一个细节都有具体原因：

细节一：为什么用显式 **`threading.Lock()`，CPython 的 GIL 不够吗？**

GIL 保证单个字节码的原子性，但 consume() 是”读取→比较→写入”三个步骤的组合。在两个线程同时调用 consume() 时，GIL 可能在”读”和”写”之间切换线程，导致两个线程都读到”还有余量”，然后都写入，budget 超支。显式锁保证这三步是原子的。

细节二：**refund()**** 方法——为什么某些工具调用不消耗 budget？**

execute_code 工具被调用时会退还一次 budget。背后的逻辑是：budget 的本质是”模型思考和决策的次数”，execute_code 只是运行了一段程序，没有经过模型推理——它是确定性的、机械的。如果把它和真正的”模型决策循环”等同计费，会导致包含大量代码执行的任务比实际更快耗尽 budget，强迫模型在代码还没跑完就急着收尾。

细节三：70%/90% 两级压力预警——为什么不直接在 100% 时停止？

预警不是单独的 system message（那样会破坏 user/assistant/tool 的消息交替顺序，导致 API 报错），而是嵌入在工具结果的 JSON 里：

模型在读取工具结果时自然看到这段信息，不产生额外的消息对。为什么分两级？70% 时提示”开始考虑收尾”，模型通常能在剩余的 30% 里平稳落地；90% 时提示”现在必须响应”，防止模型在最后几次迭代里还在调用不必要的工具。单级提示太晚，模型来不及优雅收束；单级提示太早，会导致模型在任务还没完成时就停下来。

四、为什么工具并发用 ThreadPoolExecutor 而不是 asyncio.gather

结论：Hermes 的工具大部分是同步阻塞的，在 async context 里跑同步代码需要 **`run_in_executor()`，本质还是线程池，中间多一层抽象没有意义。**

当模型在一次响应里返回多个工具调用，Hermes 用 ThreadPoolExecutor 并发执行：

_MAX_TOOL_WORKERS = 8 这个数字本身也是经过调优的。太小（比如 2）意味着多个 terminal.execute() 无法真正并行；太大（比如 32）每个线程可能持有自己的 subprocess、event loop、文件描述符，内存和句柄压力过高。8 对应”典型模型输出 2-5 个并行工具调用”的场景。

还有一个经常被忽视的细节：并发执行，顺序回填。

为什么要保持顺序？因为 LLM API 的消息结构要求 tool result 的 tool_call_id 和对应的 tool call 一一对应，且顺序要匹配。如果按完成先后顺序回填，API 可能因为顺序不一致而拒绝请求。

五、Gateway 会话状态为什么从 os.environ 迁移到 contextvars

结论：os.environ 是进程全局的，多个并发会话共享同一个 namespace，后写的值会覆盖先写的值。

早期版本用 os.environ 传递会话上下文：

这在单会话场景下没问题。但 Gateway 同时处理多个平台的消息——用户 A 发来一条消息，用户 B 在同一毫秒也发来一条。两个 agent 实例在不同线程运行，都往同一个 os.environ 写入自己的 THREAD_ID，互相覆盖，结果 A 的工具调用结果被发送到 B 的聊天窗口。

修复方案是 contextvars.ContextVar：

ContextVar 是任务局部的——每个 asyncio task（以及它派生的线程）有自己的值副本，并发的会话之间天然隔离，不需要任何锁。

这个修复还有一个细节：set_session_vars() 返回 reset tokens，handler 在 finally 块里用这些 tokens 还原旧值，防止值泄漏到后续的 task 复用场景。同时 get_session_env() 先查 ContextVar，查不到再 fallback 到 os.environ——这让 CLI 和 cron 等非并发场景（它们仍然用环境变量传值）完全不需要改动。

这是一个很典型的”用对了并发原语”的 fix：问题不在于缺少锁，而在于用了进程级作用域来存储应该是任务级作用域的状态。

六、为什么上下文压缩分两层，而不是统一一个阈值

结论：两层的触发条件和数据来源不同，强行合并会在 Gateway 长会话场景下导致每轮都触发压缩。

两层是：

  • Agent ContextCompressor（50%）：在 agent 工具循环内运行，能拿到 API 返回的精确 token 数
  • Gateway Session Hygiene（85%）：在 agent 处理消息之前运行，只能用字符数粗估 token

如果把 Gateway 层的阈值也设为 50%，会出现这个问题：Telegram/Discord 这类平台，用户可能整晚都在和 agent 对话，消息在不处理的时候继续积累。等第二天第一条消息进来时，Gateway 检查一下历史，超 50% 了，压缩一次。然后 agent 开始处理，agent 内部又检查，又超 50%，又压缩一次。一条消息触发两次压缩，而且 Gateway 层用的是粗估值，压缩边界不准。

把 Gateway 层设为 85%，它的定位就变成了”兜底防止 API 报错”，只在历史真的快撑满 context window 的时候才介入；50% 的 Agent 层负责日常的、精准的压缩。两层各司其职，没有重叠。

七、压缩算法里 protect_first_n 是硬编码的 3，为什么不让用户配置

结论：前 3 条消息（system prompt + 首轮对话）建立了整个会话的意图和上下文，如果这部分被压缩掉，摘要就会丢失”这个任务是什么”的信息，后续的重建都是错的。

消息结构通常是：

第 0 条是 system prompt，第 1 条是用户的原始请求，第 2 条是 agent 的首次响应——这三条建立了”整个任务的上下文”。如果这三条被压缩进摘要，摘要里会有类似”用户让 agent 重构认证模块”的表述，但失去了原始请求的措辞，也失去了 system prompt 里的人格设定和约束条件。后续 re-compression 更新摘要时，这些信息只会越来越模糊。

protect_last_n 是可配置的（默认 20），因为”最近多少条是活跃上下文”确实因任务性质不同而变化。但”首次建立的意图”是普遍重要的，硬编码反而是在防止错误配置。

八、子 Agent 的 Budget 为什么独立，而不是从父 Agent 扣

结论：子 Agent 干的事和父 Agent 干的事在性质上是平行的，不是包含关系。共享 budget 会导致父 Agent 因为子 Agent 的工作量而提前截停自己的主任务。

直觉上，”父有 90 次，生了个子，子用了 20 次，父只剩 70 次”看起来很合理——整个任务的总计算量有上限。但实际场景是：

父 Agent 的任务是”重构整个项目”，它委派子 Agent 去并行处理三个子模块。子 Agent 各自需要 30-40 次迭代。如果用共享 budget，父 Agent 派出三个子 Agent 之后自己基本没有余量了，没办法综合子 Agent 的结果、处理冲突、做最终的整合工作。

独立 budget 的逻辑是：子 Agent 是父 Agent 的工具调用，一次 delegate_task 在父 Agent 的视角里只算一次迭代（委派出去然后拿回结果）。子 Agent 内部跑了多少次，和父 Agent 的工作深度无关。

如果用户想要整体上限，可以通过减小 delegation.max_iterations 或者限制同时运行的子 Agent 数量（MAX_CONCURRENT_CHILDREN = 3）来间接控制总计算量。

九、SQLite 为什么用 jitter retry 而不是用数据库级锁等待

结论：SQLite 内置的 busy handler 使用固定退避，多个 writer 会以相同节奏醒来然后同时抢锁，产生 convoy effect。随机 jitter 打散重试时机，从根本上避免这个问题。

Hermes 的场景：Gateway 同时处理多条平台消息，多个 agent 线程可能同时写 state.db。SQLite 的 WAL 模式允许并发读，但写是串行的。

SQLite 自己的 timeout 参数让数据库等待指定时间后报错，内部用的是固定间隔轮询（busy handler）。想象三个 writer 同时来抢锁，一个拿到了，另外两个被 busy handler 挡回去，等 100ms，然后同时再来——每次都是同时，一直 convoy。

jitter（随机 20-150ms）让这三个 writer 醒来的时间彼此错开，每次只有一个在锁竞争窗口内，其他两个错开了，自然排队。

另一个细节：为什么用 BEGIN IMMEDIATE 而不是默认的 BEGIN？

默认 BEGIN 的模式是”推迟加锁”——开始事务时不加写锁，直到真正执行第一次写操作时才加。如果两个事务都开始了，第一个执行写操作时另一个才发现拿不到锁，但第一个已经做了一部分工作，只能回滚重做。

BEGIN IMMEDIATE 在事务开始时就加写锁。失败就失败，没有做了一半再回滚的浪费。结合 jitter，失败→等随机时间→重试，这个循环比”开始-做了一半-发现锁冲突-回滚-重来”效率高得多。

WAL checkpoint 每 50 次写入触发一次，用 PASSIVE 模式——只刷当前没有读者需要的 frames，不阻塞任何进行中的读取。这防止 WAL 文件在 Gateway 长期运行时无限增长，但不引入额外的写入延迟。

十、为什么 Prompt Cache 只打系统提示词和最近 3 条消息，不打全部历史

结论：Anthropic 每次请求最多 4 个 breakpoint，系统提示词最稳定（命中率最高），最近 3 条是滚动窗口的权衡——打更多反而没意义。

system_and_3 策略：

Anthropic prompt cache 的原理是：如果这次请求的前缀和上次完全相同，那一段 token 不重新计算，只收 25% 的费用（有 5 分钟 TTL）。系统提示词在 Hermes 里是会话级别冻结的（prompt_builder.py 构建一次后不再修改），所以第一个 breakpoint 几乎每次都能命中，节省最多。

为什么不把更多历史消息也打上 breakpoint？因为一旦消息列表增长，每次新消息来了，之前的”第 N 条”变成了”第 N+1 条”，位置变了，cache 就 miss 了。打得越多，miss 越多，反而降低命中率。

最近 3 条是一个经验值——假设两次 API 调用之间用户发了一条消息、agent 回了一条，”最近 3 条”里通常有 1-2 条和上次调用时完全相同，能命中。打 4 条以上的边际收益很低，因为 4 个 breakpoint 里那最后一个 slot 被最近 3 条占掉了，没有多余的。

还有一个实现细节很容易被忽视：

不修改原始消息是为了防止 cache_control 标记污染 history。如果原始 history 被打上了标记，下一轮构建消息时会看到上一轮的 breakpoint 残留，Anthropic API 会因为 breakpoint 顺序不对而报错。

十一、为什么有了 Agent 内部压缩，还要有 Gateway 层的 Session Hygiene

结论：Agent 内部压缩需要先启动 agent 才能运行，但 session 可能在 agent 启动前就已经超大——Gateway 层在 agent 启动之前先处理这种情况。

这个问题可以反过来问：如果只有 Agent 内部压缩，会出什么事？

场景：用户昨晚用 Telegram 和 agent 聊了两小时，历史达到 95K tokens（接近 128K 的 context window）。今天早上发来第一条消息，Gateway 开始构建 AIAgent 实例，把这 95K tokens 的历史传进去，agent 调用 _build_api_messages()，然后调 API——API 直接返回 context_length_exceeded，因为加上系统提示词和新消息已经超限了。

Agent 内部压缩的触发点是”进入主循环前”或”API 调用失败后”，但前提是 agent 实例已经创建好了。Gateway Session Hygiene 在消息刚到达、agent 还没创建时就先估算一下 token（用字符数粗估，不精确但足够），如果超 85% 就先压一次，再把压缩后的历史传给 agent。这样 agent 启动时拿到的历史是安全的。

两层的分工因此是：Agent 层负责”在 context 使用到 50% 时主动优化，保持运行效率”；Gateway 层负责”在 agent 启动前兜底，确保不因为历史过大直接崩掉”。

十二、压缩算法的摘要模板为什么长这样

结论：这个结构是为了让下一次压缩时能做”增量更新”而不是”重新摘要”。

每次压缩生成摘要，后续再次压缩时，这个摘要会被传给 LLM 并要求”更新它而不是重新写”。如果摘要是非结构化的叙述文本，”更新”的意思就很模糊——LLM 可能重新写一遍、可能只改一部分、可能把已完成的事当做”进行中”还留着。

有了固定的 section 结构，”更新”就变成了：把 In Progress 里完成的条目移到 Done；把新遇到的阻塞加进 Blocked；更新 Relevant Files 里的文件状态。每个 section 的职责清晰，LLM 知道什么信息应该放在哪、什么信息应该被淘汰。

_previous_summary 字段存在 ContextCompressor 实例上，跟着 agent 实例的生命周期走。这意味着同一个 agent 多次压缩，摘要是有记忆的——不会因为压缩了三次就变成”三次独立摘要的堆砌”，而是一份持续被更新的项目进度文档。
