五、记忆模块三层架构

五、记忆模块三层架构

结论先行：Hermes 的记忆系统解决了 LLM 无状态的本质问题——LLM 本身不记得任何东西，必须在每次请求时重建上下文。Hermes 用三层记忆分别解决三类”记住”问题：第一层（MEMORY.md/USER.md）解决”下次见面还认识你”，第二层（SQLite + FTS5）解决”找回过去做过什么”，第三层（技能文件）解决”记住怎么做”。

三层记忆架构全览

第三层技能索引的大小：技能以两级缓存的紧凑索引注入 system prompt，每个技能仅占一行（名称 + 一句话描述），不包含全文内容。LLM 看到索引后，当判断某技能与当前任务相关时，主动调用 skill_view(name) 加载完整 SKILL.md 内容，再按其中指令执行。索引大小取决于安装的技能数量：每个技能约 60-100 字符，20 个技能约 1500-2000 字符，对 system prompt 影响可控。进程内使用 LRU 缓存（_SKILLS_PROMPT_CACHE）避免每轮重新扫描文件系统；重启后通过 ~/.hermes/.skills_prompt_snapshot.json 磁盘快照（mtime/size 比对）加速恢复。

缓存失效规则：磁盘 snapshot 的有效性由 _build_skills_manifest() 判断——对每个 SKILL.md 记录 [mtime_ns, size]，只要有任意一个文件被修改/新增/删除，整个 snapshot 自动失效，触发冷启动全量重扫并覆写新 snapshot。LRU dict 最多缓存 8 个不同 (skills_dir, tools, toolsets, platform) 组合，超出后淘汰最久未用的条目（OrderedDict + LRU 策略）；多平台 gateway 同时服务时，不同平台的索引各自独立缓存，互不干扰。

第一层：内置持久记忆（MEMORY.md / USER.md）

做什么：两个 Markdown 文件分别存储 Agent 自己的笔记和用户画像。每次新会话开始时，这两个文件的内容以”冻结快照”形式拼入 system prompt，LLM 从第一条消息起就”认识”当前用户和当前环境。两个文件各有严格的字符上限（而非 token 上限，因为字符计数与模型无关，不同 tokenizer 下 token 数差异显著）：MEMORY.md 上限 2200 字符（约 800 tokens），USER.md 上限 1375 字符（约 500 tokens）。

为什么分两个文件？ 职责分离让 LLM 写入时更容易判断放哪个文件（环境事实 vs 用户偏好），也便于未来分别调整容量配置、生命周期策略。

写入流程（memory 工具）：

replace/remove 的匹配逻辑（子字符串匹配）：

  • 命中 0 条 → 返回 "No entry matched" 错误

  • 命中多条且内容相同（精确重复）→ 只操作第 0 条，安全

  • 命中多条且内容不同 → 返回 "Multiple entries matched, be more specific" 错误，并附上各条目前 80 字预览，要求 LLM 提供更精确的 old_text

去重机制：load_from_disk() 读取文件后用 dict.fromkeys() 去重（保留首次出现、去掉精确重复条目）；add() 时对精确重复做预判，直接跳过并提示 "Entry already exists"。注意：去重仅针对精确重复的完整条目，语义相似但文字不同的条目不会被合并。

容量超限时的处理：执行 add 操作时，系统预先计算新总长度；若超限，直接返回错误并提示当前用量和超出量，要求先 replace/remove 腾出空间。replace 同样做超限检查，确保替换后不超容量上限。

关键设计：冻结快照（Frozen Snapshot）

为什么用”冻结快照”而不是实时更新？如果每次写入记忆都重建 system prompt，Anthropic prefix cache 就会失效，导致每轮都全量计费（通常 2000~8000 token）。冻结快照保证 system prompt 在整个会话内稳定，prefix cache 持续命中，最多可将缓存命中部分的输入 token 成本降低约 90%（Anthropic prefix cache 命中按原价 10% 计费）。实际节省幅度取决于 cache 命中率，系统设计目标是让 system prompt 前缀在整个会话内保持不变以最大化命中。

本次会话写入的记忆，当前会话能立即看到吗？ 不能——但这是设计意图而非缺陷：磁盘立即持久化（原子写），LLM 在工具响应结果里能看到最新条目，而 system prompt 快照不变；下次 AIAgent.__init__() 时重新读取文件，新内容才进入快照。

安全扫描（防止提示注入攻击）：每条记忆写入前都会经过安全扫描，匹配以下威胁模式的内容将被拒绝写入。扫描范围不限于下方展示的 3 条规则，还包括凭据泄露、SSH 后门等模式，以及不可见 Unicode 字符检测。

多 profile 下的记忆隔离：记忆文件路径通过 get_memory_dir() 动态解析（调用 get_hermes_home() 读取 HERMES_HOME 环境变量），不同 profile 的 HERMES_HOME 各不相同，因此各 profile 的记忆文件完全隔离，互不干扰：

自动记忆 Flush（防止信息在压缩中丢失）

做什么：在上下文压缩前，让 LLM 主动决定哪些信息值得长期保存，避免重要细节随中间轮次的摘要被淡化甚至丢失。

触发条件：

  • 上下文压缩前自动触发（min_turns=0，强制执行）
  • 周期性轮次触发（每隔 N 轮对话，LLM 主动复盘是否有新知识值得保存；N 可通过 config.yaml 配置，默认 6 轮，设为 0 可禁用）

flush 调用的辅助 LLM 如何配置？ flush_memories() 优先通过 call_llm(task="flush_memories") 调用辅助 LLM，模型由 config.yaml 的 auxiliary.flush_memories 节点决定。若未配置，系统走 auto 提供商自动选择逻辑（通常选当前可用的较廉价模型）；若辅助 LLM 不可用，回退为主提供商直接调用，以 temperature=0.3、max_tokens=5120 发起请求。

如果 LLM 判断没有值得保存的信息，会怎么做？ 什么都不做，静默完成。flush 提示词只是请求”如果有就保存”，LLM 若认为无新知识，就不调用 memory 工具，flush_memories() 收到的 tool_calls 为空，直接进入 finally 清理 sentinel 并返回。即便 LLM 不写任何东西，flush 调用只多一次辅助 LLM 请求（较廉价），不会有额外副作用。

sentinel 标记的技术细节：sentinel 是格式为 __flush_{id(self)}_{time.monotonic()} 的唯一字符串（如 __flush_140234567890_12345.678901）。id(self) 确保不同 AIAgent 实例之间不冲突，time.monotonic() 确保同一实例的多次 flush 之间不冲突。之所以用字符串比较而非对象 identity（is），是因为 messages 列表可能经过序列化/反序列化（dict copy、JSON 往返），对象 identity 不可靠，字符串比较更稳健。

第二层：会话历史检索（session_search）

做什么：将所有历史对话存入 SQLite + FTS5 全文搜索索引，让 LLM 在遇到”我上次怎么解决这个问题”时，通过 session_search 工具主动检索过去的对话，而不需要用户重复说明背景。存储内容为完整原始消息，不截断，不摘要——append_message() 直接将 content 字段原文写入，旧 session 封存后消息一行不动，始终可搜。唯一可能截断的情况是超长工具结果（默认 100000 字符上限，超出后加 [TRUNCATED] 标记，属极端情况）。

压缩后的 session 链表：每次上下文压缩时，旧 session 被封存（end_session()），新 session 通过 parent_session_id 链接父会话，形成单向链表。封存后的旧 session 消息一行不动，session_search 的全库搜索可以命中所有历史 session 的原始内容——即使已经历多次压缩，第一个 session 的完整原始消息仍然可搜。

对话数据会永久保存吗？有自动清理吗？ 会话数据默认永久保留，没有自动 TTL 清理。用户需手动调用清理命令（只删除已结束的会话，进行中的 active session 不受影响；被删除的 session 的 child sessions 会被孤立化而非级联删除）：

多平台（CLI、Telegram、Discord）之间的记忆是共享还是隔离的？ 默认共享（同一 profile 下所有平台读取相同的 HERMES_HOME）。sessions 表中有 source 字段区分来源，session_search 默认跨所有 source 搜索，也可指定 source 过滤。若要隔离，为不同平台使用不同 profile（hermes -p telegram_bot gateway start）即可做到记忆、历史、配置完全独立。

外部记忆为什么注入 user 消息而非 system prompt？

外部记忆的 recall 结果是动态的（每次请求不同），若注入 system prompt 则每轮都会破坏 Anthropic prefix cache。注入到当前轮 user 消息末尾，system 前缀保持稳定。

外部 Provider 与内置记忆是并行运行还是互斥？ 两者并行运行，但外部 Provider 互斥注册：内置记忆（BuiltinMemoryProvider）始终先注册、不可移除；外部 Provider 任意时刻最多允许一个（尝试注册第二个会发出警告并被拒绝，以防止工具 schema 膨胀和记忆后端冲突）。两种记忆同时对 LLM 可见，各司其职：内置记忆通过 memory 工具由 LLM 主动调用，注入 system prompt 冻结快照；外部 Provider 通过 prefetch/sync_turn 后台运行，注入 user 消息末尾动态上下文。

完整生命周期时序图

各轮次记忆行为详解

结论先行：首轮和后续轮次在记忆行为上有三处关键差异——外部 Provider prefetch 首轮为空、SQLite 首轮无可用历史（但 session_search 随时可调）、queue_prefetch 首轮后才开始预热。下表和逐轮时序图给出完整说明。

机制 第 1 轮（首轮） 第 2 轮 第 N 轮（稳态）
外部 Provider prefetch 空（无上轮 queue_prefetch 缓存） 返回第 1 轮结束时 queue_prefetch 预热的结果 返回上一轮结束时预热的结果（完整缓存命中）
SQLite FTS5（session_search） 可调用（检索历史 session）；本轮消息尚未写入，当前轮不可搜 第 1 轮消息已持久化，可搜；LLM 主动调用时才触发 前 N-1 轮消息全部可搜；LLM 按需调用
_persist_session（写入 SQLite） 本轮结束后写入；写入前当前轮消息不可搜 本轮结束后增量写入（仅追加新消息） 同上，增量追加
内置记忆（MEMORY.md） 已注入 system prompt（冻结快照），LLM 从第一条消息起即可读取 system prompt 不变；LLM 写入的新记忆仍不在快照中 同上；新记忆下次会话才进快照
外部 Provider sync_turn（写入） 本轮结束后后台线程写入；第 2 轮 prefetch 可能已含本轮内容（取决于写入速度） 同上 同上（后台异步，不阻塞主流程）

关于”首轮不会调用 SQLite FTS5”：这个说法需要拆分理解。session_search 是工具，首轮 LLM 完全可以主动调用它检索历史 session——只要 LLM 判断”需要回忆过去”就会触发。首轮”没有”的，是当前轮自身的消息在 SQLite 中不可搜（因为 _persist_session 在轮后才执行）；以及外部 Provider prefetch 返回空（因为没有上轮的 queue_prefetch 缓存）。这是两件完全不同的事。
