六、上下文压缩

LLM 的上下文窗口是有限的。一个长期运行的 Agent（处理复杂编码任务、多步骤研究）在数百轮对话后，messages 列表的 token 总量会轻松超过 100K，触碰模型的硬限制。

Hermes 的上下文工程要解决的核心问题是：在有限的上下文窗口里，如何最大化保留对”继续完成任务”最重要的信息。

答案是中部压缩：任务的起点（用户目标）和最近的进展（尾部）都要保留，被压缩掉的是中间那些已经完成的工具调用历史——它们的具体内容已经不再重要，只需要一份结构化摘要告诉 LLM”我们之前做了什么”。

上下文压缩全流程

flowchart TD
    A["每次工具执行完成后检查"] --> B{token 计数 ≥\ncontext_length × 50%?}

    B -->|否| C["继续循环，不压缩"]
    B -->|是| D["⟳ compacting context…\n提示用户"]

    D --> E

    subgraph E["ContextCompressor.compress()"]
        E1["步骤 1：工具输出裁剪\n（无 LLM 调用，低成本预处理）\n对非尾部旧工具结果（>200字符）替换占位符\n效果：通常减少 30~50% token"]

        E1 --> E2

        E2["步骤 2：划定保护区"]

        subgraph E2a["消息列表分区"]
            H["头部保护（protect_first_n=3）\nsystem 首轮背景"]
            M["中部（待压缩）\n历史 tool calls\n中间轮对话"]
            T["尾部保护（~20K tokens）\n最近若干轮对话\n最新进展"]
        end

        E2 --> E3["步骤 3：用辅助 LLM 生成结构化摘要\n（廉价/快速模型，非主模型）"]

        E3 --> E4["步骤 4：重组 messages\n头部 + 摘要消息 + 尾部\n通常从 200 条 → 20~40 条"]

        E4 --> E5["步骤 5：工具调用完整性修复\n_sanitize_tool_pairs()\n补齐孤儿 tool_call/result 对"]
    end

    E --> F["步骤 6：重建 system prompt\n压缩 = 新会话\n重载最新 MEMORY.md 快照"]

    F --> G["将压缩后的 messages\n写入新的 SQLite session"]

结构化摘要：信息的精华提炼

压缩时调用辅助 LLM 生成的摘要，遵循一个固定模板，保证信息的完整性：

为什么用这个模板，而不是让 LLM 自由发挥？

自由发挥容易产生”文学性摘要”，听起来流畅但缺少具体信息（文件路径、命令、错误信息等）。结构化模板强制 LLM 保留这些”不能丢失的细节”，让下一个 context 里的 Agent 可以真正接续工作，而不是重复劳动。

迭代式摘要更新

flowchart LR
    A["第一次压缩"] --> B["生成摘要 v1\n存储在 _previous_summary"]
    B --> C["继续对话 50 轮"]
    C --> D["第二次压缩触发"]
    D --> E{有 _previous_summary?}
    E -->|有| F["将 v1 摘要作为输入\n生成迭代更新摘要 v2\n'Based on previous summary...\nnew developments...'"]
    E -->|无| G["从零生成摘要"]
    F --> H["存储 v2，丢弃 v1"]

这样可以保证多次压缩时，信息不是简单累加，而是滚动更新，保持摘要的精炼。

压缩完之后可能存在哪些问题

压缩会切割消息列表，可能在边界处产生”孤儿”——一个 tool_call 的结果消息被切掉了，或者一个 tool_result 对应的调用消息被摘要替换了。OpenAI 和 Anthropic API 都要求 tool_call / tool_result 必须成对出现，否则报 400 错误。

  flowchart TD
    A["压缩后的 messages"] --> B["_sanitize_tool_pairs()"]

    B --> C["扫描所有 assistant 消息\n收集 surviving_call_ids 集合"]
    B --> D["扫描所有 tool 消息\n收集 result_call_ids 集合"]

    C --> E{对比两个集合}
    D --> E

    E --> F["孤儿 tool_result\n（result 有，call 没有）\n→ 删除此 tool 消息"]
    E --> G["孤儿 tool_call\n（call 有，result 没有）\n→ 在 call 后面插入存根结果\n'[Result from earlier conversation —\n see context summary above]'"]

如何查看被压缩的内容？

压缩后原始历史并没有被删除，只是不在 active messages 里了。有两种方式找回：

方式 1：直接问 Agent（LLM 能看到摘要）

摘要消息本身就在 messages 里，LLM 下一轮即可读到。直接对话：

方式 2：查 SQLite + JSON 文件（完整原始记录）

压缩时，旧 session 在 SQLite 里被打上 "compression" 标记并结束，同时创建新 session id。完整原始对话保存在：

  • SQLite：~/.hermes/sessions.db，可通过 /history 命令检索
  • JSON 文件：~/.hermes/logs/session_<旧session_id>.json，包含压缩前的完整 messages

原始记录永远不会被物理删除——压缩只影响 active messages（当前 API 调用传入的 messages 列表），不影响持久化存储。/history 命令或直接读 JSON 文件即可回溯所有历史对话。
