十五、完整大纲

Hermes Agent 完整大纲

本文档是 Hermes Agent 系列文章的导读索引，梳理各章节的核心内容与相互关系，帮助读者快速定位感兴趣的模块。

系列文章目录

一、整体架构概览

以文件依赖链为主线，讲解 Hermes Agent 的五层结构：工具注册层（tools/registry.py）、工具实现层（tools/*.py）、工具编排层（model_tools.py）、Agent 核心层（run_agent.py）、接入层（cli.py、gateway/、acp_adapter/）。介绍项目整体定位——多平台 AI Agent 框架，以及贯穿全局的三条设计原则：工具即插件、平台无关核心、缓存友好架构。

二、Agent 核心实现

详解 run_agent.py 中 run_conversation() 的完整 ReAct 执行流程：构建系统提示 → 进入 while 循环（最多 max_iterations 次）→ 调用 LLM API（OpenAI 格式，支持多 provider）→ 解析响应（有 tool_calls 则执行工具并追加结果继续循环，无则返回最终响应）→ 迭代预算与 token 预算双重限制 → 上下文压缩触发时机（50% 阈值）。

三、工具系统：注册、发现与调用

注册机制：registry.register() 接收 name、toolset、schema、handler、check_fn、requires_env，统一管理工具元数据。工具发现：启动时按序 import 所有工具文件，触发各文件末尾的 registry.register() 调用。Toolset 过滤：_HERMES_CORE_TOOLS 定义核心工具集，支持 enabled_toolsets / disabled_toolsets 参数按需裁剪。调用链路：handle_function_call() → 查 registry → 执行 handler → 返回 JSON 字符串。Agent 级工具（todo、memory）在 run_agent.py 中拦截，不经过 handle_function_call()。

四、技能系统：注册、发现与调用

技能是存放在 ~/.hermes/skills/<name>/SKILL.md 的 Markdown 文件，通过斜杠命令注入为用户消息（而非系统提示），以保护 prefix cache 不被破坏。渐进式披露：L0（技能描述，极短）、L1（技能摘要，中等长度）、L2（技能全文，完整操作指南）。技能 vs 工具：工具是代码（Python 函数），技能是知识（Markdown 文档）；技能告诉 Agent “如何做”，工具给 Agent “做的能力”。

五、记忆模块三层架构

介绍 Hermes Agent 的记忆体系：工作记忆（当前对话 messages 列表）、会话记忆（SessionDB，SQLite + FTS5 全文检索，跨会话持久化）、长期记忆（~/.hermes/memory/，用户偏好与知识沉淀）。讲解记忆的写入时机、检索策略，以及记忆与上下文压缩的协同关系。

六、上下文工程

触发条件：当对话历史 token 数超过模型上下文窗口的 50% 时自动触发压缩（agent/context_compressor.py）。中部压缩策略：保留最早的若干轮（锚定初始任务）和最近的若干轮（保持当前状态），压缩中间部分为摘要，避免丢失任务起点和最新进展。辅助 LLM（agent/auxiliary_client.py）：压缩摘要由独立的辅助 LLM 生成，与主 Agent 解耦，可配置为更小、更快的模型以控制成本。

七、Prompt 拼接机制

系统提示组装（agent/prompt_builder.py）：按固定顺序拼接——角色定义 → 工具说明 → 记忆 → 上下文文件 → 技能内容。顺序固定是 prefix cache 有效的前提。Anthropic Prompt Caching（agent/prompt_caching.py）：在系统提示末尾和长对话历史的固定位置插入 cache_control: {"type": "ephemeral"} 标记，利用 prefix cache 机制大幅降低重复 token 费用。核心约束：缓存建立后禁止修改已缓存内容（不能重载记忆、不能切换工具集、不能重建系统提示），唯一例外是上下文压缩。

八、子 Agent 委托（delegate_task）

delegate_task 工具允许主 Agent 创建子 Agent 实例，将独立子任务并行或串行委托执行，子 Agent 拥有独立的对话历史和工具集。全局状态隔离：_run_single_child() 在子 Agent 执行前后保存/恢复 _last_resolved_tool_names 进程全局变量，防止子 Agent 的工具解析结果污染父 Agent。适用场景：大任务拆解、并行信息收集、隔离高风险操作。

九、MCP 客户端集成

tools/mcp_tool.py（~1050 行）实现完整的 MCP（Model Context Protocol）客户端，支持动态发现和调用 MCP Server 暴露的工具。凭据池与错误分类恢复：MCP 连接支持凭据池轮换，对错误进行分类（可重试 vs 不可重试），实现自动恢复，提升长时间运行的稳定性。工具命名：MCP 工具以 server_name/tool_name 格式注册，避免与内置工具命名冲突。

十、危险命令审批与检查点回滚

危险命令检测（tools/approval.py）：在执行前扫描命令，匹配危险模式（rm -rf、dd、mkfs 等），触发用户审批回调。检查点机制：在执行高风险操作前保存系统状态快照，支持操作失败后回滚，降低不可逆操作的风险。审批回调：通过 callbacks 机制注入平台特定的审批 UI（CLI 交互确认、Gateway 消息推送等）。

十一、execute_code（程序化工具调用）

execute_code 工具提供安全的代码执行沙箱，支持 Python 等多种语言。讲解沙箱隔离机制、执行超时控制、输出捕获与截断策略，以及与终端工具（terminal）的区别——execute_code 适合结构化计算任务，terminal 适合系统命令与文件操作。

十二、智能路由、凭据池与错误恢复

智能路由（agent/models_dev.py）：基于 models.dev 注册表实现 provider-aware 的模型路由，根据模型能力、上下文长度、成本自动选择最优 provider。凭据池：多组 API Key 轮换使用，单个 Key 限流时自动切换，提升高并发场景下的可用性。错误分类恢复：将 API 错误分为可重试（限流、超时）和不可重试（认证失败、参数错误），分别采用指数退避重试和快速失败策略。

十三、终端多环境、ACP 适配器与 Gateway 平台

终端多环境（tools/environments/）：支持 local、docker、ssh、modal、daytona、singularity 六种执行环境，通过统一接口抽象，Agent 无需感知底层差异。ACP 适配器（acp_adapter/）：提供 VS Code / Zed / JetBrains 集成，将 Hermes Agent 嵌入 IDE 工作流。Gateway 平台（gateway/）：支持 Telegram、Discord、Slack、WhatsApp、HomeAssistant、Signal，统一主循环处理消息分发、斜杠命令、会话持久化，平台差异由各 adapter 封装。

Hermes Agent 核心实现：工程师为什么这么做

从工程决策视角解读 Hermes Agent 的设计取舍：为什么工具用自注册而非配置文件？为什么技能注入为用户消息而非系统提示？为什么上下文压缩选择中部压缩而非截断？为什么禁止中途重建系统提示？每个决策背后的约束与权衡。

三条核心设计原则

原则一：工具即插件。工具通过 registry.register() 自注册，框架无需感知具体工具实现。新增工具只需三步：建文件、加 import、加 toolset。

原则二：平台无关核心。run_agent.py 不依赖任何平台细节，CLI / Gateway / ACP 通过 callbacks 注入平台特定行为（审批、通知、输入），核心循环保持纯净。

原则三：缓存友好架构。系统提示结构固定、技能以用户消息注入、禁止中途重建提示——所有设计决策都服务于 Anthropic prefix cache 的最大化命中率，直接影响运行成本。
